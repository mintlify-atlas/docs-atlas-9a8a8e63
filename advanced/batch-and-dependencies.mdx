PA_PVP supports multi-item batches where each item carries its own mode, previous state, evidence, and dependency declarations. The batch header sets shared policy for the entire run; individual items can override specific fields. Each item is processed independently within the batch, but dependency and resource declarations create cross-item relationships that the kernel resolves deterministically — without asking for operator input when conflicts arise.

<Tip>
  For simple single-item runs, you don't need a batch header or item wrappers. Plain language input is sufficient. Use formal wrappers when you need deterministic batching, explicit modes, dependencies, per-item previous state, or machine-to-machine exchange.
</Tip>

## Batch header syntax

The optional batch header sets shared policy for all items in the run:

```text
<<<B keep_open=NO expire_days=30 exec_capability=NO_RUNTIME close_policy=SIM_OK ssot_scale=MIN resource_pool=ENG:2,MONEY:1>>>
```

| Field | Description |
|---|---|
| `keep_open` | When `YES`, disables expiry for all items (no item transitions to EXPIRED). Per-item `keep_open` overrides this. |
| `expire_days` | Default inactivity expiry in days for items in STANDBY. Per-item `expire_days` overrides this. Fallback is 30 days if neither is set. |
| `exec_capability` | `NO_RUNTIME` (default) means the protocol does not assume real execution capability; steps are planned and simulated. `RUNTIME_OK` means real execution is available. |
| `close_policy` | `SIM_OK` (default) allows simulated closure when explicitly labeled. `REAL_ONLY` requires `HISTORICAL`, `REAL`, or `EXPERIMENT` evidence to close any item. |
| `ssot_scale` | `MIN` (default) emits compact item structure; `DEBUG` emits full diagnostic structure for all items. See [SSOT scale](#ssot-scale) below. |
| `resource_pool` | Declared concurrency capacity as `RID:CAPACITY` pairs (e.g., `ENG:2,MONEY:1`). Used to resolve scheduling conflicts. |

## Item wrapper syntax

Each item in a batch is wrapped with an `<<<I>>>` block:

```text
<<<I id=E1 title="short title" keep_open=NO expire_days=30 depends_on=E2 uses=ENG:1 reversibility=REVERSIBLE>>>
...payload...
<<<END>>>
```

| Field | Description |
|---|---|
| `id` | **Required.** Unique identifier for this item within the batch. Missing `id` results in `INVALID_ITEM` — that item is skipped without blocking the rest of the batch. |
| `title` | Short human-readable label for the item. |
| `keep_open` | Per-item override of the batch `keep_open` setting. `YES` disables expiry for this item only. |
| `expire_days` | Per-item override of the batch `expire_days` setting. |
| `depends_on` | Space-separated list of item ids this item depends on (e.g., `depends_on=E2 E3`). These are execution-order constraints. |
| `uses` | Resource units this item consumes when active (e.g., `uses=ENG:1`). Must reference ids declared in the batch `resource_pool`. |
| `reversibility` | `REVERSIBLE`, `COSTLY_REVERSIBLE`, or `IRREVERSIBLE`. Affects closure policy, hedge eligibility, and AskUser activation rules. |

## Dependency model

`depends_on` declarations create **execution-order constraints**, not automatic verdict blockers. The protocol resolves them as follows:

- If `depends_on` is non-empty and any listed dependency is not `CLOSED` or `DISCARDED`, the dependent item remains ping-pong OPEN and Step 1 priority applies
- Step 1 for the dependent item is either: the earliest non-conflicting step to resolve the blocking dependency, OR an explicit waiver of the dependency (which opens Falsification Debt)
- Non-waived dependent steps **must not execute** before their dependencies resolve — the protocol does not assume the dependency will close on its own

**Circular dependencies:** If a cycle is detected among items in the same batch (including self-dependencies), every item in the cycle receives:

- `verdict = DO LATER`, `new_state = STANDBY`
- `trigger = CIRCULAR_DEPENDENCY`

The cycle is treated as a structural modeling error, not a timing problem. Step 1 must break the cycle by doing at least one of: splitting the item into a decision item and an execution item, removing or replacing a `depends_on` edge, explicitly waiving a dependency (debt applies), or discarding a dominated item in the cycle. Cycle-breaking steps are recorded in `[STEPS]` as planned actions for when the item is revived.

## Resource concurrency

Declare a shared resource pool on the batch header and per-item usage to enable concurrency management:

```text
<<<B resource_pool=ENG:2,GPU:1>>>
<<<I id=E1 uses=ENG:1,GPU:1>>>
<<<I id=E2 uses=ENG:1>>>
<<<I id=E3 uses=ENG:2>>>
```

Resolution rules:

1. Consider only items that would be `DO NOW` (`new_state=ACTIVE`) before resource rules are applied
2. For each resource `RID`, sum the units requested by all ACTIVE items
3. If demand exceeds capacity for any `RID`, demote ACTIVE items in **reverse priority order** until all overloads clear:
   - Keep highest-priority items ACTIVE first: `impact HIGH > MED > LOW`, then `urgency HIGH > MED > LOW`, then `id` lexical order as a stable tie-break
   - Demoted items: `verdict = DO LATER`, `new_state = STANDBY`, `trigger = DEPENDENCY_BLOCK`
   - Step 1 for demoted items: wait for resource availability, or split/sequence execution to reduce demand

Only use resource ids that were declared in the batch `resource_pool`. The protocol does not invent new resource ids or units.

## Batch coverage rule

<Warning>
  Every input id must appear in `[QUEUE]` **and** in an emitted `[ITEM id=...]` block. No input id may be omitted from either location. No extra id may appear unless it was introduced by an explicit split in this run.

  If this coverage invariant is violated, the **entire output is marked `INVALID`** with `InvalidReason: BATCH_COVERAGE` and `CoverageDiff` listing the missing and extra ids. An INVALID output is non-operational — do not execute its steps, recover its state, or use it as `<<<PREV>>>`.
</Warning>

## Batch enforcement rules

| Rule | Behavior |
|---|---|
| Item missing `id` | `INVALID_ITEM` — that item is skipped; the rest of the batch continues normally |
| Duplicate `id` in same batch | `REJECT_SECOND` — second occurrence is DISCARDED with `trigger=DUPLICATE_ID_REJECTED`; no auto-merge |
| Circular dependency | All items in the cycle → `DO LATER`, `trigger=CIRCULAR_DEPENDENCY` |
| Resource overload | Lower-priority items demoted → `DO LATER`, `trigger=DEPENDENCY_BLOCK` |

## SSOT scale

`ssot_scale` controls how much structure is emitted per item in the output:

- **`ssot_scale=MIN`** (default): emits compact item structure (`[ITEM_PANEL]`, `[DERIVED]`, `[STEPS]`, `[NEXT]`) for items without `<<<PREV>>>`. However, if an item has `contested=YES` or `dominant_gate != NONE`, it automatically expands to full DEBUG structure for that item regardless of the batch-level `MIN` setting.

- **`ssot_scale=DEBUG`**: always emits full item structure for all items, including `[DECISION_MODEL]`, `[EVIDENCE_BINDING]`, `[DATA_ACQUISITION]`, `[ALTERNATIVE]`, `[PROBE]`, `[PROBE_LOG]`, and `[REVIEW]`.

Scaling changes only the emitted output — it does not change the SSOT semantics or routing rules. A snapshot produced with `ssot_scale=MIN` is still a valid machine state for the next run; omitted sections are treated as absent, not contradictory.
